# 第二阶段 Spec：团队协作能力增强

## 1. 背景

第一阶段已经完成本地交付闭环：Leader 单入口、LangGraphJS 编排、角色适配层、会议节点、repo/runner/storage/artifacts 能力和基础验证链路均已具备。

第二阶段不应急于做 Web 工作台或云端执行，而应增强“团队协作真实性和稳定性”，让系统从“能串行交付”升级为“能根据任务特征组织更可靠的研发协作”。

## 2. 阶段目标

第二阶段目标：

> 让 Leader 不只是串行调用角色，而是能基于任务特征选择协作路径，并让会议、回流、风险和上下文沉淀形成稳定机制。

## 3. 范围

### 3.1 包含范围

- 更细的角色职责和输入/输出协议。
- 更标准化的团队会议机制。
- 更完整的任务状态、失败回流和争议处理。
- 更稳定的上下文摘要、artifact 索引和阶段报告。
- 更明确的风险分级和审批触发规则。
- 更可靠的长流程恢复和重复执行保护。

### 3.2 不包含范围

- Web 工作台。
- 多用户系统。
- 云端执行环境。
- GitHub / GitLab 深度集成。
- 真正复杂的并行调度。
- 自动发布和生产环境操作。

## 4. 推荐能力模块

### 4.1 角色协议增强

为 PM / Architect / Developer / QA 定义稳定协议：

- 输入：任务摘要、相关 artifact、当前状态、风险标记。
- 输出：结构化结论、置信度、风险、下一步建议、是否需要老板确认。
- 失败：明确原因、所需补充信息、建议回流状态。

### 4.2 会议机制增强

会议从“结构化总结”升级为“可复用决策机制”：

- 明确会议触发条件。
- 固定角色发言顺序。
- 记录争议点和裁决理由。
- 输出可执行 action items。
- 标记是否需要老板拍板。

### 4.3 回流机制增强

补齐以下回流场景：

- 测试失败：QA -> Developer -> QA。
- 方案冲突：Developer / Architect -> Meeting。
- 需求变化：Leader -> PM -> Planning。
- 风险升高：任一角色 -> Leader -> Owner decision。

### 4.4 上下文和产物沉淀

增强 artifact 管理：

- artifact 类型更细化。
- 每个状态保存上下文摘要。
- 每次角色运行可追踪输入与输出。
- 最终报告能引用关键 artifact，而不是只给文本总结。

### 4.5 风险与审批

定义风险分级：

- low：Leader 可直接推进。
- medium：需要 Architect / QA 标记并在报告中说明。
- high：进入会议或老板审批。

审批触发应覆盖：

- 范围变化。
- 高风险命令。
- 破坏性操作。
- 验收标准变化。
- 多方案且影响交付方向。

## 5. 验收标准

第二阶段完成后，应满足：

1. Leader 能基于任务特征选择普通路径、会议路径、审批路径或阻塞路径。
2. 角色输出遵循统一结构，且测试覆盖角色协议。
3. 会议输出包含争议点、裁决理由、风险和 action items。
4. 测试失败、方案冲突、需求变化都有明确回流路径。
5. artifact 和上下文摘要可支撑长流程恢复。
6. 风险分级和审批触发有测试覆盖。
7. CLI 入口继续兼容第一阶段命令。
8. `npm run typecheck`、`npm test`、`npm run build` 通过。

## 6. 推荐实施顺序

1. 固化角色输入/输出协议。
2. 升级 meeting 输出结构。
3. 增强风险分级和审批触发。
4. 补齐回流场景。
5. 强化 artifact/context 索引。
6. 做全链路测试和阶段报告。
