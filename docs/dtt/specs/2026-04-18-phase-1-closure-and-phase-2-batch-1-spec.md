# 第一阶段真实收口与第二阶段 Batch 1 启动 Spec

## 1. 背景

当前仓库的路线图和阶段完成说明都把第一阶段标记为“已完成”，但最新一次 fresh verification 显示基线尚未真正收口：

- `npm run typecheck` 通过
- `npm run build` 通过
- `npm test` 失败，失败点为 `tests/leader.test.ts` 中“meeting 识别阻塞时进入 blocked 并写入暂停信息”

这说明第一阶段虽然在能力设计上已成立，但当前仓库状态还不能作为“已验证完成”的稳定基线。

在此基础上，用户已确认采用路径 **B**：

> 先把第一阶段真正收口，再直接进入第二阶段的第一批开发。

## 2. 目标

本次工作的目标分为两部分：

### 2.1 第一阶段真实收口

- 修复当前阻碍第一阶段收口的验证失败
- 恢复“typecheck / test / build”三项验证全绿
- 让第一阶段“已完成”重新具备 fresh verification 证据支撑

### 2.2 第二阶段 Batch 1 启动

- 在第一阶段重新收口后，立刻进入第二阶段的首批开发
- 首批开发聚焦“协作基础稳定化”，而不是产品化工作台或外部平台集成
- 让第二阶段从组织协作质量提升开始，而不是从界面或平台能力开始

## 3. 约束与边界

### 3.1 必须保持

- CLI 单入口继续兼容：`start`、`resume`、`approve`、`resolve-block`
- Leader 继续作为唯一外部入口
- 状态流转、审批触发、安全边界继续由系统规则控制
- 阶段收口必须以 fresh verification 为准，而不是只沿用旧文档结论

### 3.2 本次不做

- Web 工作台
- 多用户系统
- GitHub / GitLab 深度集成
- 云端执行环境
- 自动 Git commit / PR 流程编排
- 让大模型直接绕过 workflow 执行命令或写文件

## 4. 方案比较

### 方案 A：只修第一阶段，不进入第二阶段

优点：最稳，范围最小。

缺点：不能满足用户已明确选择的路径 B，也无法把当前修复直接转化为第二阶段起点。

### 方案 B：先收口第一阶段，再启动第二阶段的协作基础批次（推荐）

优点：

- 能先恢复稳定基线，再在稳定基线上推进第二阶段
- 与既有路线图一致：第二阶段先做协作能力增强，不抢跑工作台或平台化
- 能把当前暴露出来的路由/会议/阻塞一致性问题转化为第二阶段第一批基础建设内容

缺点：

- 范围比单纯修 bug 更大，需要同时处理收口和第二阶段第一批交付

### 方案 C：先收口第一阶段，再直接进入真实大模型接入

优点：能更快验证模型价值。

缺点：

- 当前仓库刚暴露出流程一致性回归，先接入真实模型会放大调试难度
- 与第二阶段既有 plan 不一致；现阶段更适合先稳住协议、路由、会议和风险边界

## 5. 推荐方案

采用 **方案 B**。

也就是说：

1. 先把第一阶段重新收口到“当前可验证完成”的状态；
2. 再直接开始第二阶段 **Batch 1：协作基础稳定化**；
3. 真实大模型接入仍属于第二阶段范围，但放在协议、会议、风险和回流机制更稳定之后，而不是本次首批实现的核心目标。

## 6. 本次执行范围

### 6.1 第一阶段收口范围

- 复现并定位当前 failing test 的根因
- 修复 planning / meeting / blocked 路由与测试期望之间的不一致
- 确保第一阶段相关行为在测试层面有明确且可解释的约束
- 重新运行：
  - `npm run typecheck`
  - `npm test`
  - `npm run build`
- 如有必要，更新阶段完成说明，使其与 fresh verification 结果一致

### 6.2 第二阶段 Batch 1 范围

本批次不做完整第二阶段，而是做“第二阶段第一批可落地基础”。

重点包括：

- 把角色协议、会议结论、阻塞/审批/回流路径之间的关系进一步固化
- 消除当前代码、测试、阶段文档之间的行为理解偏差
- 为后续风险规则集中化与真实大模型接入保留稳定边界
- 形成一个清楚的“第二阶段已启动”批次交付，而不是只停留在路线图层面

建议优先落点：

- 路由一致性：planning 与 meeting 对 blocked / owner decision 的处理规则明确化
- 协议一致性：角色输出、会议输出、状态转换使用统一决策信号
- 测试一致性：为关键路径补齐或修正测试，避免文档说完成、验证却不通过
- 阶段记录：明确写出“第一阶段已重新收口，第二阶段 Batch 1 已启动”的文档产物

## 7. 验收标准

### 7.1 第一阶段收口验收

- `npm run typecheck` 通过
- `npm test` 通过
- `npm run build` 通过
- 当前仓库行为与第一阶段文档口径一致，不再出现“文档已完成、fresh verification 不通过”的冲突

### 7.2 第二阶段 Batch 1 启动验收

- 至少有一批实际代码/测试/文档交付，能说明第二阶段已经开始，而不是只停留在计划
- Leader / meeting / blocked / approval 的关键协作路径更清晰、更稳定
- CLI 行为不退化
- 有明确文档说明：第二阶段从协作基础稳定化开始，而不是从工作台或平台化开始

## 8. 验证方式

本次完成前至少执行：

- `npm run typecheck`
- `npm test`
- `npm run build`

如涉及 CLI 兼容性，再补充 smoke：

- `npm run dev -- start "..."`
- `npm run dev -- approve <taskId>`
- `npm run dev -- resume <taskId> --note "..."`
- `npm run dev -- resolve-block <taskId> --note "..."`

## 9. 预期交付物

- 第一阶段收口修复代码
- 对应测试修复或补充
- fresh verification 结果
- 第二阶段 Batch 1 启动相关代码/测试/文档
- 一份能够说明“第一阶段已真正收口、第二阶段已启动”的阶段记录

## 10. 本次不做的决定

本次不把“真实大模型接入”作为首批实现主目标。

原因不是放弃大模型，而是为了避免在阶段基线尚未重新收口时，把不稳定因素进一步放大。真实大模型接入仍属于第二阶段，但应建立在更稳定的协作协议、会议机制、风险边界和测试基线上。
