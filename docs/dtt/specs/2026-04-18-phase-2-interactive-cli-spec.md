# 第二阶段收尾：持续交互 CLI Spec

## 1. 背景

当前 DevTeamOS 已经具备第二阶段中后段的大部分基础能力：

- 角色协议稳定化
- 会议 / 审批 / 阻塞 / 回流机制
- 受控真实模型接入
- OpenAI / MiniMax provider 扩展
- `.env` 配置支持

但目前老板或操作者与系统交互，仍然主要依赖一组离散命令：

- `start`
- `resume`
- `approve`
- `resolve-block`

这意味着系统虽然已经有暂停、审批、阻塞、恢复等协作能力，但在使用体验上仍然是“单步命令式”的，而不是“持续交互式”的。

用户明确提出：

> 下一阶段想先做持续交互的 CLI，能够在一个会话里持续提示阻塞、审批、补充输入等。

随后用户又明确判断：

> 这个方向应该算**第二阶段**，而不是第三阶段。

## 2. 结论定位

本次将“持续交互 CLI”定义为：

> **第二阶段的收尾能力**，而不是第三阶段产品化工作台的替代品。

原因是：

- 它解决的是“人机交互闭环是否顺畅”的问题，本质仍属于协作流程与使用入口增强。
- 它能把已经存在的暂停、审批、阻塞、恢复能力真正变成连续可用体验。
- 它还不是图形化工作台，不应直接视为第三阶段开始。

## 3. 本次目标

本次希望实现的是：

1. 提供一个**持续交互式 CLI 模式**，让用户不必手动频繁复制 `taskId` 和切换命令。
2. 在一个会话里完成：
   - 发起任务
   - 查看当前状态
   - 处理澄清
   - 处理老板审批
   - 处理阻塞恢复
   - 直到完成或显式退出
3. 保持现有 `start / resume / approve / resolve-block` 命令兼容，不破坏脚本化调用。

## 4. 方案比较

### 方案 A：把现有离散命令包装成一个交互式 REPL【推荐】

做法：

- 新增一个交互命令（例如 `interactive`）
- 在单个 CLI 会话中，根据任务状态自动决定调用：
  - `runLeaderTask`
  - `resumeLeaderTask`
  - `approveLeaderTask`
  - `resolveBlockedTask`
- 用户主要通过自然输入或少量交互选择完成流程

优点：

- 对现有状态机与任务模型侵入最小
- 能最大程度复用现有能力
- 最适合作为第二阶段收尾增强

缺点：

- 仍是文本交互，不是工作台

### 方案 B：大幅升级当前 CLI，默认所有命令都进入交互模式

优点：

- 统一入口更强

缺点：

- 容易破坏当前已有 CLI 兼容性
- 会影响脚本化与测试稳定性

### 方案 C：直接跳到第三阶段工作台

优点：

- 最终形态更直观

缺点：

- 与当前“第二阶段收尾”目标不符
- 会过早进入第三阶段产品化成本

## 5. 推荐方案

采用 **方案 A**：

> **在保留现有离散命令的前提下，新增一个持续交互式 CLI 模式。**

这样可以把第二阶段的流程能力真正串起来，但又不会破坏现有脚本式入口。

## 6. 范围

### 6.1 包含范围

- 新增持续交互 CLI 入口（命令名可在 plan 中确定）
- 在一个会话中串起：
  - 初始任务输入
  - clarifying 补充
  - owner decision 批准
  - blocked 解除阻塞
  - done 完成
- 会话中可展示当前状态、等待原因、下一步建议
- 保留现有离散命令兼容

### 6.2 不包含范围

- 不替代第三阶段工作台
- 不做图形化 UI / TUI 框架
- 不做实时 token streaming
- 不改核心 workflow 规则
- 不新增新的审批、阻塞或角色语义

## 7. 设计原则

### 7.1 持续交互是现有状态机的“外壳”，不是新状态机

持续交互 CLI 只负责更顺畅地驱动现有任务状态：

- `clarifying`
- `awaiting_owner_decision`
- `blocked`
- `done`

它不定义第二套流程规则。

### 7.2 保持脚本式命令兼容

现有：

- `start`
- `resume`
- `approve`
- `resolve-block`

继续保留，持续交互 CLI 是增强入口，而不是破坏性替换。

### 7.3 聚焦“会话闭环”，不过度承诺“实时观感”

本次持续交互 CLI 的重点是：

- 一个会话不中断地处理下一步动作
- 提示当前为什么暂停
- 提示用户应该做什么

而不是：

- 完整实时日志流
- 图形化进度面板
- 多窗口并发任务管理

### 7.4 第二阶段收尾，不提前进入第三阶段

这项能力完成后，项目仍然只是：

- 让 CLI 交互更连续

而不是：

- 已经有产品化工作台

## 8. 交互目标体验

理想体验示例：

1. 用户进入交互模式
2. 输入需求
3. CLI 执行到暂停点时，直接提示：
   - 当前状态
   - 暂停原因
   - 可选操作
4. 用户在同一会话中输入补充说明、批准或解除阻塞
5. 任务继续推进
6. 最终输出完成摘要或继续等待下一步

## 9. 验收标准

完成后至少应满足：

1. 用户可以在一个 CLI 会话里完成至少一条完整任务闭环。
2. clarifying / awaiting_owner_decision / blocked 三类暂停点都能在同一会话里被正确提示与处理。
3. 当前状态、等待原因、下一步动作对用户清晰可见。
4. 现有 `start / resume / approve / resolve-block` 命令继续可用。
5. `npm run typecheck`、`npm test`、`npm run build` 通过。

## 10. 负路径验收

除正向流程外，还应覆盖：

1. 用户输入无效操作时，CLI 能给出清晰提示，而不是崩溃。
2. 任务在 blocked / awaiting_owner_decision / clarifying 切换时，会话不会丢失上下文。
3. 退出交互模式后，任务仍可通过现有离散命令继续恢复。

## 11. 风险与控制

### 风险 1：把交互 CLI 做成半个工作台

控制：

- 只做文本会话式闭环
- 不做图形化与复杂展示层

### 风险 2：破坏现有离散命令兼容性

控制：

- 新增交互入口，而不是替换现有命令
- 保持原有 CLI 输出格式与测试

### 风险 3：交互层偷偷改写 workflow

控制：

- 交互层只调用现有 leader API
- 不新增第二套状态迁移逻辑

## 12. 对路线图的影响

本次工作会把第二阶段从“能力已经具备”推进到“使用体验真正连贯”，因此属于：

> **第二阶段收尾能力**

而不是第三阶段产品化工作台本身。
