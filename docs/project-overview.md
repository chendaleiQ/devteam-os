# DevTeamOS 项目总览

## 1. 项目定义

DevTeamOS 要做的不是单个 coding agent，而是一支能真实交付结果的 AI 研发团队。

老板只负责提出目标、回答少量关键问题和做最终决策；`Leader` 负责对外沟通、对内调度；其他角色像真实团队一样分工协作，在后台 7x24 持续推进任务。

一句话定义：

> DevTeamOS 是一个由 Leader 负责对外、由专业角色协同执行、可持续交付结果的 AI 研发团队。

## 2. 核心价值

- 让老板从“盯过程”切换为“提目标、看结果、做关键决策”
- 用 `Leader -> 团队` 的组织模型降低直接管理多个 Agent 的成本
- 通过 7x24 持续执行显著提升研发推进效率
- 用审批、回流和审计机制控制自动化交付风险

## 3. 组织模型

### 老板

- 提出目标或需求
- 回答澄清问题
- 审批高风险决策
- 接收阶段汇报和最终交付

### Leader

- 作为唯一对外窗口和任务负责人
- 判断需求是否清晰
- 决定是否拉 PM 联合澄清
- 决定是否召开团队会议
- 组织 Architect、Developer、QA 等角色执行任务
- 在关键节点向老板汇报并请求决策

### PM

- 协助澄清需求
- 输出范围边界和验收标准
- 识别模糊点和隐含需求

### Architect

- 分析影响范围
- 提出技术方案和风险
- 给出任务拆分建议

### Developer

- 负责实际实现
- 对代码、配置、联调问题给出修改结果
- 遇到阻塞时反馈给 Leader

### QA

- 制定验证策略
- 执行测试并判断风险
- 推动失败回流和修复

### DevOps

- 在后续阶段负责部署、发布检查和回滚准备

## 4. 协作机制

### 4.1 Leader 负责制

默认情况下，老板不直接面对所有角色，而是只和 Leader 沟通。Leader 负责解释任务、协调资源、推动执行、管理节奏并输出汇报。

### 4.2 按需拉人

PM 只在需求澄清时重点参与；Architect、Developer、QA 根据任务阶段介入；复杂场景由 Leader 决定是否拉更多角色参与。

### 4.3 团队会议

遇到复杂需求、高风险改动、方案冲突或需要统一结论时，Leader 可以发起团队会议。会议输出必须包含：

- 结论
- 风险
- 下一步行动
- 是否需要老板确认

### 4.4 7x24 持续推进

团队应默认持续执行，不受工作时间限制。只有在以下情况才暂停等待老板：

- 需求必须补充信息
- 需要老板审批高风险决策
- 团队建议改变目标或范围

## 5. 端到端流程

1. 老板向 Leader 提出目标
2. Leader 判断是否需要澄清或召开会议
3. PM / Architect / Developer / QA 按需参与分析
4. Leader 组织执行并持续推进
5. QA 完成验证并推动回流修复
6. Leader 向老板汇报结果、风险和建议
7. 老板做接受、补充修改或暂停的决定

## 6. 产品边界

### 当前要做的

- `Leader -> 团队` 的真实协作模型
- 需求澄清、会议、执行、测试、汇报闭环
- 本地代码仓库读写与测试验证
- 老板离线时的持续推进能力

### 当前不优先做的

- 多用户和组织管理
- 完整 Web 工作台
- 自动生产发布
- 复杂多仓库和大规模并行调度

## 7. 成功标准

如果老板可以像给真实团队下任务一样提出目标，而系统能由 Leader 带着团队持续推进到可验证交付，并在需要时再把问题带回老板决策，就说明 DevTeamOS 的方向是成立的。
