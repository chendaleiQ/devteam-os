# DevTeamOS 项目总览

## 1. 项目定义

DevTeamOS 当前的产品定义，不再是“自研完整 AI 研发团队执行平台”，而是：

> **一个由 `Leader` 对外负责、对任务进行治理、并可接入外部执行器完成真实开发工作的 AI 研发任务治理层。**

一句话定义：

> DevTeamOS 是一个以 `Leader` 为入口、以审批与风险治理为核心、以外部执行器为执行底座的 AI 研发任务治理层。

## 2. 核心价值

- 让老板从“盯过程”切换为“提目标、看结果、做关键决策”
- 用 `Leader` 统一任务入口，减少老板直接管理多个 agent 或平台的负担
- 用审批、回流、审计和 artifact 沉淀控制自动化交付风险
- 在不自研完整执行栈的前提下，复用现成 agent 平台能力

## 3. 组织模型

### 老板

- 提出目标或需求
- 回答澄清问题
- 审批高风险决策
- 接收阶段汇报和最终交付

### Leader

- 作为唯一对外窗口和任务负责人
- 判断需求是否清晰
- 决定是否需要澄清、会议、审批或阻塞恢复
- 将任务派发给外部执行器，并统一回收结果
- 在关键节点向老板汇报并请求决策

### 执行角色（逻辑层）

- PM / Architect / Developer / QA 仍然是系统里的**逻辑角色语义**
- 这些角色不再要求由 DevTeamOS 自己实现为内部 LLM agent
- 它们可以来自外部执行器返回的阶段产物、计划、PR、验证结果或摘要

### 外部执行器

- 外部执行器负责实际开发执行，例如：
  - Devin
  - Open SWE
  - GitHub Copilot cloud agent
  - OpenHands
- DevTeamOS 不把自己继续定位为这些执行能力的底层替代品

## 4. 协作机制

### 4.1 Leader 负责制

默认情况下，老板不直接面对所有角色，而是只和 Leader 沟通。Leader 负责解释任务、协调资源、推动执行、管理节奏并输出汇报。

### 4.2 按需拉人

治理层只保留角色语义和任务路径选择，不再要求所有角色都由仓库内部代码直接实现。复杂场景下，`Leader` 可以：

- 请求外部执行器提供计划、实现、验证结果
- 汇总多方结论形成会议纪要
- 在高风险节点向老板发起审批

### 4.3 团队会议

遇到复杂需求、高风险改动、方案冲突或需要统一结论时，Leader 可以发起治理层意义上的“会议汇总”。会议输出必须包含：

- 结论
- 风险
- 下一步行动
- 是否需要老板确认

### 4.4 7x24 持续推进

系统仍可持续推进，但真正的开发执行由外部执行器承担。只有在以下情况才暂停等待老板：

- 需求必须补充信息
- 需要老板审批高风险决策
- 团队建议改变目标或范围

## 5. 端到端流程

1. 老板向 Leader 提出目标
2. Leader 判断是否需要澄清、会议、审批或阻塞处理
3. Leader 将任务派发给外部执行器
4. 外部执行器完成计划、实现、验证或 PR 产出
5. Leader 汇总结果、识别风险、决定是否回流或请求老板决策
6. Leader 向老板汇报结果、风险和建议
7. 老板做接受、补充修改、审批或暂停的决定

## 6. 产品边界

### 当前要做的

- `Leader` 统一入口与治理模型
- 澄清、审批、阻塞、回流、汇报闭环
- 结构化 artifact、checkpoint、delivery report
- 外部执行器的统一接入与编排
- 老板离线时的持续推进能力

### 执行真实性准则

- 运行时只允许接入真实外部执行器，不再保留 `mock-executor`、fake executor 或伪造执行结果路径
- “先让项目可用”指打通真实 `OpenHands` 链路，而不是增加 mock 逃生口
- 验证执行器链路时，不允许用 fake fixture 冒充真实执行完成

### 当前不优先做的

- 自研 LLM provider
- 自研 repo runtime / command runner
- 自研 patch proposal 协议
- 自研完整多角色执行平台
- 多用户和组织管理
- 完整 Web 工作台

## 7. 成功标准

如果老板可以只和 `Leader` 沟通，而系统能在接入外部执行器的情况下，持续推进任务、控制风险、沉淀结构化结论，并在需要时只把关键决策带回老板，就说明 DevTeamOS 的方向是成立的。
