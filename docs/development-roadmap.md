# DevTeamOS 全项目阶段性目标与路线图

本文档是 DevTeamOS 整个项目开发的主路线图，用于说明项目从原型验证到组织化能力建设的阶段性目标、推进顺序与验收口径。它不是某一阶段的临时讨论记录，而是面向项目整体的正式文档。

## 总体判断标准

DevTeamOS 的建设顺序不是先堆满功能，而是按“先验证真实交付闭环，再增强协作能力，再完成产品化与标准化”的路径推进。

每一阶段都应回答同一个问题：

> 这一阶段，是否让 DevTeamOS 更接近一个能稳定交付结果的 AI 研发团队？

## 阶段验收 Checklist

勾选表示该阶段已达到对应验收口径，可作为后续阶段推进的判断依据。

- [x] **第一阶段：Leader 驱动的本地交付原型**
  - 老板只与 Leader 沟通即可发起与推进任务
  - 系统可在本地仓库内完成分析、修改、验证与汇报，形成最小可用交付闭环
  - 团队角色协作已能支撑真实需求推进，并可在老板离线时持续推进任务

- [ ] **第二阶段：团队协作能力增强**
  - Leader 能根据任务类型选择合适的协作路径，而不只是固定串行调用角色
  - 会议、方案分歧、测试失败、需求变化等关键场景具备明确且可复用的处理机制
  - 关键协作结论能沉淀为结构化产物，长时任务在中断、失败或等待审批后可恢复推进
  - 整体协作表现应体现为一个有分工、有回流、有升级路径的团队系统

- [ ] **第三阶段：产品化工作台**
  - 老板可通过工作台发起任务、查看进度、处理审批，而不必依赖 CLI
  - 工作台可清晰展示任务状态、阶段结论、关键决策、代码变更与测试结果
  - Leader 汇报有统一展示入口，历史任务可追踪、可回看、可定位关键结论
  - 对外演示时，工作台足以支撑完整任务流程展示，而不需要大量口头补充解释

- [ ] **第四阶段：仓库与执行环境标准化**
  - 系统可在标准化环境中稳定处理真实项目仓库，而不只是演示仓库
  - 代码变更能够进入标准分支与 PR 流程，并与现有团队协作方式兼容
  - 执行环境具备明确边界、隔离和审计能力，常见研发动作可重复、可恢复、可追踪
  - 团队可将 DevTeamOS 作为现有工程流程的一部分，而不是额外的实验性工具

- [ ] **第五阶段：高级协作与组织能力**
  - 系统能够支持多任务并行推进与更复杂的团队协作结构
  - 组织层面的争议处理、质量评估、复盘与持续运营能力成熟可用
  - DevTeamOS 从任务执行器升级为具备持续协作、持续交付与组织管理能力的 AI 研发组织系统

## 第一阶段：Leader 驱动的本地交付原型

### 阶段定位

验证最核心的产品假设：老板只和 Leader 沟通，系统能在本地仓库内完成分析、修改、验证与汇报，形成最小可用交付闭环。

### 当前状态

**已于 2026-04-18 重新收口，并通过 fresh verification（typecheck / test / build）。**

### 本阶段成立意味着

- Leader 可以成为统一任务入口
- 团队角色协作可以支撑真实需求推进
- 系统可以对本地代码仓库执行实际修改与验证
- 任务可在老板离线时持续推进，遇到审批或澄清节点再回到老板

## 第二阶段：团队协作能力增强

### 阶段目标

在第一阶段闭环成立后，把“能跑通”升级为“能更稳定、更像团队地跑通”，提升协作真实性、回流质量和长时执行能力。

### 当前状态

**已启动，当前已进入第二阶段收尾：在 Batch 1：协作基础稳定化与受控真实模型接入 / provider 扩展之后，补齐了持续交互 CLI。**

当前阶段不是跳到第三阶段产品化，而是在第二阶段内部继续推进：先完成协作基础稳定化，再把真实模型接入限制在受控边界内验证价值，并在收尾阶段补齐持续交互 CLI，让系统能在同一会话内处理澄清、审批与阻塞恢复。

- Batch 1 已完成的基础：角色协议、会议结论、阻塞/审批/回流路径与 `riskSignals` 已完成基础对齐。
- 当前新增的中后段能力：默认仍走 `mock`，其中 PM / Architect / QA 已走受控 mock provider 输出链路；显式配置真实 provider 后，当前支持 **OpenAI + MiniMax**。其中 MiniMax 已统一到官方 Quickstart 口径：`DEVTEAM_LLM_PROVIDER=minimax`、`DEVTEAM_LLM_MODEL=MiniMax-M2.7`、`ANTHROPIC_BASE_URL=https://api.minimaxi.com/anthropic`、`ANTHROPIC_API_KEY=<key>`。PM / Architect / QA 复用既有结构化角色输出链路；Developer 默认无 `llm` 配置时仍保留兼容摘要路径，显式 `mock` / `openai` / `minimax` 时可生成结构化 `patch proposal` 并经系统校验后受控写入。
- 当前新增的收尾能力：持续交互 CLI 已可在一个会话内承接澄清、审批和阻塞恢复，属于第二阶段协作闭环补强，不是第三阶段工作台。
- 当前仍未开放：`delete` / `rename` / 直接自由写文件 / 直接执行命令等高风险能力。

### 重点建设

- 更细的角色分工与协作路径选择
- 更标准化的会议机制与结构化结论沉淀
- 更完整的任务状态流转、回流与升级机制
- 更稳定的上下文管理、产物沉淀与断点恢复能力
- 更清晰的风险分级、审批触发与异常处理

### 当前批次范围（Batch 1：协作基础稳定化）

- 统一角色输出、会议结果与状态转移使用的决策信号
- 让 `riskSignals` 开始在关键协作路径中真实生效
- 稳定 meeting / blocked / owner decision / resume 之间的基础关系
- 保持 CLI 单入口和第一阶段兼容

### 当前推进位置（第二阶段收尾：持续交互 CLI + 既有受控能力收口）

- 建立 `llm` provider 抽象，默认 `mock`，显式 real
- 当前真实 provider 支持 **OpenAI + MiniMax**
- MiniMax 启用口径已按官方 Quickstart 收口到 `DEVTEAM_LLM_PROVIDER=minimax`、`DEVTEAM_LLM_MODEL=MiniMax-M2.7`、`ANTHROPIC_BASE_URL=https://api.minimaxi.com/anthropic`、`ANTHROPIC_API_KEY=<key>`；旧 `MINIMAX_API_KEY` / `MINIMAX_BASE_URL` 不再作为文档主路径
- PM / Architect / QA 已可走同一套受控角色输出协议，默认 mock、显式可切 OpenAI / MiniMax
- Developer 已可在显式 `mock` / `openai` / `minimax` provider 下输出结构化 `patch proposal`，并通过系统校验与受控写入进入后续验证链路
- 持续交互 CLI 已纳入第二阶段收尾范围：同一 CLI 会话内可继续处理澄清、审批与阻塞恢复
- 上述能力仍然是 CLI 内的协作增强，不是第三阶段的产品化工作台或图形化产品

### 当前批次暂不包含

- 产品化工作台
- 外部平台集成
- 完整长流程治理能力
- `delete` / `rename` / 直接自由写文件 / 直接执行命令等高风险能力
- **第三阶段产品化工作台**：定位不变，当前尚未进入；不应因持续交互 CLI 或真实模型接入而误判阶段前移

### 完成标准 / DoD / 验收要点

- Leader 不再只是固定串行调用角色，而是能根据任务类型选择合适的协作路径
- 会议、方案分歧、测试失败、需求变化等关键场景，有明确且可复用的处理机制
- 关键协作结论能沉淀为结构化产物，而不是散落在一次性对话中
- 长时任务在中断、失败或等待审批后，可以继续恢复推进，而不是从头开始
- 整体表现应让外部观察者感受到这不是单 agent 执行，而是一个有分工、有回流、有升级路径的团队系统

## 第三阶段：产品化工作台

### 阶段目标

把当前原型升级为可演示、可使用、可持续运营的产品化工作台，让老板和团队成员不必依赖命令行即可完成主要操作。

### 重点建设

- 统一任务工作台
- Leader 汇报与进度展示
- 会议记录、决策结论与关键产物展示
- 代码变更、测试结果与审批信息展示
- 历史任务追踪与检索能力

### 完成标准 / DoD / 验收要点

- 老板可以通过工作台发起任务、查看进度、处理审批，而不必依赖 CLI
- 工作台能够清晰呈现任务状态、阶段结论、关键决策、代码变更与测试结果
- Leader 对老板的汇报有统一展示入口，信息表达足够清楚，适合直接用于管理沟通
- 历史任务可追踪、可回看、可定位关键结论，避免系统成为“黑箱”
- 对外演示时，工作台足以支撑完整任务流程展示，而不需要大量口头补充解释

## 第四阶段：仓库与执行环境标准化

### 阶段目标

让 DevTeamOS 能更稳定、更安全地接入真实研发环境，使交付结果能够直接进入团队现有研发流程。

### 重点建设

- GitHub / GitLab 等代码平台集成
- 分支、提交、PR、评论与合并流程支持
- 标准化执行环境与任务隔离机制
- 更清晰的命令、文件、权限与审计边界
- 面向真实仓库的稳定接入与运行保障

### 完成标准 / DoD / 验收要点

- 系统可以在标准化环境中稳定处理真实项目仓库，而不是只适合演示仓库
- 代码变更能够进入标准分支与 PR 流程，并与现有团队协作方式兼容
- 执行环境具备明确边界、隔离和审计能力，能支持更安全的落地使用
- 常见研发动作具备可重复、可恢复、可追踪的执行方式，减少环境差异带来的不确定性
- 团队可以把 DevTeamOS 作为现有工程流程的一部分，而不是额外的实验性工具

## 第五阶段：高级协作与组织能力

### 阶段目标

从“能够交付单个任务”升级为“像一个可持续运转的 AI 研发组织”，支持更复杂的协作结构、任务并行和组织治理。

### 重点建设

- 多任务并行推进
- 多 Leader 或多团队协作模式
- 更复杂的会议、争议裁决与协同机制
- 指标体系、质量评估与组织级复盘
- 更强的发布、回滚与持续运营能力

### 阶段结果

到这一阶段，DevTeamOS 不再只是一个任务执行器或工作台，而是一个具备持续协作、持续交付与组织管理能力的 AI 研发组织系统。

## 阶段顺序与当前重点

项目建议按以下顺序推进：

1. 第一阶段：Leader 驱动的本地交付原型
2. 第二阶段：团队协作能力增强
3. 第三阶段：产品化工作台
4. 第四阶段：仓库与执行环境标准化
5. 第五阶段：高级协作与组织能力

当前重点仍然是第二阶段内部能力建设，而不是提前进入第三阶段。现阶段以第一阶段已重新收口并重新验证通过为前提，先稳住协议、会议、风险规则与回流边界，再在此基础上推进受控真实模型接入，并在第二阶段收尾补齐持续交互 CLI；默认路径仍是 mock，其中 PM / Architect / QA 已进入受控 provider 链路，真实 provider 当前支持 OpenAI / MiniMax，且都只在显式配置下启用。持续交互 CLI 只表示 CLI 内的协作闭环更完整，不代表第三阶段产品化工作台已经开始。
