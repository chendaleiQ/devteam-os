# DevTeamOS 技术架构

## 1. 架构目标

技术架构需要支撑四件事：

- `Leader -> 团队` 的组织模型
- 老板离线时的 7x24 持续执行
- 在澄清、审批和高风险动作上的可暂停、可恢复
- 对真实代码仓库的可验证交付

第一阶段不追求复杂平台化，重点是把执行闭环做对。

## 2. 总体结构

第一阶段建议采用 `TypeScript + Node.js`，整体可以拆成 6 层：

1. 入口层：CLI 或对话入口，负责接收老板需求
2. Leader 编排层：负责判断路径、调度角色、控制节奏
3. 角色层：PM、Architect、Developer、QA
4. 工作流层：管理状态、回流、暂停和恢复
5. 执行层：仓库读写、命令执行、测试运行
6. 存储层：任务状态、产物、日志、检查点

## 3. 执行模型

### 3.1 Leader 是主控

第一阶段不做“所有 Agent 自由协商”的复杂模式，而是由 Leader 作为主控：

- 接收老板输入
- 判断是否需要澄清
- 判断是否需要会议
- 决定调用哪个角色
- 汇总结果并继续推进

### 3.2 串行为主，会议作为特殊节点

第一阶段默认采用串行执行，而不是复杂并行。这样更容易控制：

- 上下文一致性
- 错误回流路径
- 审批暂停点
- 长流程恢复

团队会议不等于真正的多线程并发，而是一个结构化汇总节点：

- 先收集各角色意见
- 再由 Leader 汇总为结论
- 结论进入后续执行

### 3.3 人工等待点最少化

系统默认持续执行，只有在以下情况进入等待：

- `clarifying`：老板必须补充信息
- `awaiting_owner_decision`：老板必须审批或拍板
- `blocked`：外部条件不足，无法安全继续

## 4. 状态机

第一阶段建议使用轻量状态机，状态如下：

- `intake`
- `clarifying`
- `planning`
- `meeting`
- `developing`
- `testing`
- `reporting`
- `awaiting_owner_decision`
- `done`
- `blocked`

关键回流：

- `intake -> clarifying`
- `planning -> meeting`
- `testing -> developing`
- `reporting -> awaiting_owner_decision`
- `awaiting_owner_decision -> planning | developing | done`

## 5. 核心数据对象

建议至少有以下结构化对象：

### Task

任务主对象，记录：

- 任务标题
- 老板原始需求
- 当前状态
- 当前负责人，始终为 Leader
- 关联仓库路径

### Artifact

阶段产物，例如：

- 需求澄清记录
- 验收标准
- 技术方案
- 会议纪要
- 测试报告
- 最终交付报告

### AgentRun

记录一次角色执行：

- 角色名
- 输入摘要
- 输出摘要
- 开始和结束时间
- 是否成功

### StateTransition

记录状态变化：

- 从哪个状态切换
- 切换到哪个状态
- 触发原因
- 触发时间

### ApprovalRequest

记录老板需要决策的事项：

- 决策问题
- 背景说明
- 候选方案
- 推荐方案
- 当前状态

### ToolCall

记录工具调用：

- 使用了什么工具
- 操作了哪些路径或命令
- 结果是否成功
- 是否触发风险标记

## 6. 模块划分

建议第一阶段按如下模块组织：

- `leader/`：Leader 的决策和调度逻辑
- `agents/`：PM、Architect、Developer、QA 的角色实现
- `workflow/`：状态机、回流、暂停和恢复
- `meeting/`：会议输入收集与结论汇总
- `repo/`：仓库搜索、读取、变更摘要
- `runner/`：命令执行与测试运行
- `artifacts/`：产物生成与持久化
- `storage/`：任务状态、日志、检查点
- `cli/`：老板入口和交互命令

## 7. 工具权限边界

不同角色可用工具应该有边界，而不是全部开放。

### Leader

- 可读任务状态、产物、汇总信息
- 可发起角色调用和流程切换
- 不直接执行大规模代码修改

### PM / Architect

- 可读仓库和文档
- 可输出结构化分析
- 不直接写代码或执行高风险命令

### Developer

- 可搜索仓库
- 可修改文件
- 可运行低风险验证命令

### QA

- 可运行测试命令
- 可读取日志和结果
- 不直接修改业务代码

### 高风险操作

以下动作默认不允许自动执行，或必须显式审批：

- 破坏性 Git 操作
- 生产环境发布
- 高风险数据迁移
- 权限逻辑的大范围变更

## 8. 7x24 持续执行设计

为了真正体现 AI 团队效率，系统必须支持长时间连续运行。

### 8.1 检查点

每完成一个关键阶段就保存检查点，包括：

- 当前状态
- 当前上下文摘要
- 已产生产物
- 最近一次工具调用结果

### 8.2 恢复继续

如果任务中断，系统应能基于检查点恢复，而不是从头开始。

### 8.3 等待机制

当进入 `clarifying` 或 `awaiting_owner_decision` 时，系统暂停执行并保留完整上下文，等待老板回复后继续。

### 8.4 阶段汇报

长任务执行过程中，Leader 应能生成阶段性汇报，而不是只在结束时输出一次结果。

## 9. 推荐目录结构

```text
src/
  cli/
  leader/
  agents/
  workflow/
  meeting/
  repo/
  runner/
  artifacts/
  storage/
  schemas/
```

## 10. 当前技术风险

当前最需要注意的问题有：

- 长流程上下文过长导致输出漂移
- 测试失败后的修复循环可能失控
- 命令执行边界如果不收紧，存在安全风险
- 团队会议如果没有固定结构，容易变成冗长总结

因此第一阶段要优先保证：

- 状态清晰
- 工具边界清晰
- 检查点可恢复
- 交付结果可验证
